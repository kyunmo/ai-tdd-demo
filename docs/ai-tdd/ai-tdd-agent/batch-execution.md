# 배치 실행 가이드

> 다중 클래스의 테스트를 한 번에 생성할 때의 실행 전략, 순서 결정, 진행 관리 규칙을 정의합니다.

---

## 1. 트리거 조건

다음과 같은 요청을 받으면 배치 모드로 실행합니다.

**활성화 키워드**:
- "전체 테스트 생성", "패키지 테스트 생성", "폴더 전체 테스트"
- "service/ 테스트 생성해줘", "모든 서비스 테스트"
- "일괄 테스트 생성"

**요청 예시**:
```
"service 패키지 전체 테스트 생성해줘"
"UserService, UserController, UserMapper 테스트 만들어줘"
"src/main/java/com/nhcard/al/tt/ 아래 전체 테스트 생성"
```

## 2. 대상 수집 알고리즘

### 2.1. 패키지/폴더 단위 요청

```
입력: 패키지 경로 또는 폴더 경로

1. Glob으로 대상 파일 수집:
   Glob: {지정경로}/**/*.java

2. 제외 대상 필터링:
   - *Application.java (메인 클래스)
   - *Config.java, *Configuration.java (설정 클래스)
   - *Dto.java, *Request.java, *Response.java, *VO.java (데이터 클래스)
   - *Exception.java (예외 클래스)
   - *Constants.java (상수 클래스)

3. 테스트 대상 확정:
   - @Service, @Component → 포함
   - @RestController, @Controller → 포함
   - @Mapper → 포함
   - *Util, *Helper (static 메서드 보유) → 포함
   - 판별 불가 → 사용자에게 확인

4. 결과: 대상 클래스 목록 + 각 클래스 유형
```

### 2.2. 명시적 클래스 목록 요청

```
입력: "UserService, UserController, UserMapper"

1. 각 클래스명에 대해 소스 파일 탐색 (SKILL.md §3.2 알고리즘)
2. 모두 찾으면 → 목록 확정
3. 일부 못 찾으면 → 찾은 것만 목록화 + 못 찾은 것 보고
```

## 3. 실행 순서 결정

### 3.1. 의존성 기반 순서

클래스 간 의존 관계를 분석하여 **하위 의존성부터** 생성합니다.

```
순서 결정 알고리즘:

1. 각 클래스의 생성자 파라미터(의존성) 추출
2. 의존성 그래프 생성
3. 위상 정렬(Topological Sort)로 실행 순서 결정

기본 레이어 우선순위 (의존성 분석 불가 시):
  Utility → Mapper → Service → Controller

이유:
  - Util: 의존성 없음, 가장 먼저 생성
  - Mapper: DB 접근만, 다른 계층에 의존하지 않음
  - Service: Mapper/Util에 의존
  - Controller: Service에 의존
```

### 3.2. 순서 결정 예시

```
입력: UserController, UserService, UserMapper, MaskingUtil

의존성 분석:
  MaskingUtil     → 의존성 없음
  UserMapper      → 의존성 없음
  UserService     → [UserMapper, MaskingUtil]
  UserController  → [UserService]

실행 순서:
  1. MaskingUtil (Utility, 의존성 없음)
  2. UserMapper (Mapper, 의존성 없음)
  3. UserService (Service, 1·2에 의존)
  4. UserController (Controller, 3에 의존)
```

### 3.3. 순환 의존성 처리

```
순환 의존성 발견 시:
  1. 순환 그룹 내 클래스를 레이어 우선순위로 정렬
  2. 정렬 불가 시 → 알파벳순
  3. 사용자에게 순환 의존성 존재를 경고
```

## 4. 배치 실행 프로세스

### 4.1. 전체 흐름

```
[대상 수집] → [순서 결정] → [진행 확인] → [순차 생성] → [종합 리포트]

각 클래스별로 SKILL.md의 3.1~3.7 단계를 반복 실행합니다.
```

### 4.2. 진행 전 사용자 확인

대상 수집과 순서 결정 후, 실행 전에 사용자에게 계획을 제시합니다.

```
## 배치 테스트 생성 계획

**대상**: {N}개 클래스
**예상 생성 테스트**: 약 {N × 10~15}개

| 순서 | 클래스명 | 유형 | 기존 테스트 | 예상 테스트 수 |
|---|---|---|---|---|
| 1 | MaskingUtil | Utility | 없음 | ~10개 (신규) |
| 2 | UserMapper | Mapper | 없음 | ~10개 (신규) |
| 3 | UserService | Service | 3개 있음 | ~8개 (추가) |
| 4 | UserController | Controller | 없음 | ~12개 (신규) |

진행하시겠습니까?
```

### 4.3. 순차 실행 및 중간 보고

```
각 클래스 완료 시 중간 보고:

  [{순서}/{전체}] {클래스명}Test 완료
  - 생성: {N}개 테스트
  - 컴파일: PASS
  - 테스트 실행: PASS ({N}개 성공)
  - 다음: {다음 클래스명}
```

### 4.4. 실패 시 처리

```
개별 클래스 실패 처리:

1. 컴파일/테스트 실패 → SKILL.md의 재시도 정책 적용 (최대 3회)
2. 3회 재시도 후에도 실패 →
   a. 해당 클래스를 SKIP으로 표시
   b. 실패 사유 기록
   c. 다음 클래스로 진행 (나머지 중단하지 않음)
3. 의존 클래스 실패 시 →
   a. 해당 클래스에 의존하는 후속 클래스도 경고 표시
   b. 후속 클래스는 실행하되, 실패할 수 있음을 명시
```

## 6. 규모별 실행 전략

### 6.1. 소규모 (5개 이하)

```
- 순차 실행
- 각 클래스별 중간 보고
- 완료 후 종합 리포트
```

### 6.2. 중규모 (6~20개)

```
- 레이어별 그룹화하여 실행
  1차: Utility 전체
  2차: Mapper 전체
  3차: Service 전체
  4차: Controller 전체
- 레이어 완료 시 중간 리포트
- 전체 완료 후 종합 리포트
```

### 6.3. 대규모 (20개 초과)

```
- 사용자에게 우선순위 확인 요청
- 핵심 도메인부터 실행 권장
- 레이어별 + 도메인별 2차원 그룹화
- 10개 단위로 중간 리포트
- 실패율이 높으면 (30% 이상) 중단하고 원인 분석 권고
```

## 7. 주의사항

1. **전체 빌드 먼저 확인**: 배치 시작 전 `./gradlew compileJava` 성공 확인
2. **기존 테스트 보존**: 추가 모드가 기본, 기존 테스트는 절대 삭제하지 않음
3. **중단 가능**: 사용자가 중단 요청 시 즉시 중단하고 진행 상황 보고
4. **독립적 검증**: 각 클래스 테스트는 독립적으로 컴파일/실행 검증
5. **누적 회귀**: 마지막에 `./gradlew test`로 전체 테스트 수트 실행하여 회귀 없음 확인
