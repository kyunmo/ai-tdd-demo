### 1.2. 자율 실행 동작 흐름

```
Step 1: 개발자 요청
├── IntelliJ에서 Claude Code 실행
└── "UserService 테스트 생성" 입력

Step 2: 소스 파일 탐색 (B1 알고리즘)
├── Glob: src/main/java/**/{ClassName}.java
├── 0건 → 확장 검색 src/main/**/{ClassName}.java
├── 2건+ → .claude.md 기본 패키지로 필터
└── 결과: 소스 파일 경로 확정

Step 3: 소스 분석 → 구조화된 결과 (B4)
├── 클래스 유형 판별 (어노테이션 > 클래스명 > 패키지)
├── 의존성 목록 추출 → @Mock 대상
├── public 메서드 시그니처 추출
├── throw 문, if/else 분기 카운트
└── 도메인 로직 식별 (암호화, 마스킹, 감사로그)

Step 4: 테스트 수 산출 (B2 공식)
├── 총 테스트 수 = max(10, public_methods × 3)
├── Level 1 = public_methods
├── Level 3 = throw_statements
├── Level 4 = max(2, domain_checks)
└── Level 2 = 총 - L1 - L3 - L4

Step 5: 기존 테스트 확인 (B5 로직)
├── Glob: src/test/java/**/{ClassName}Test.java
├── 존재 → 기존 @Test 추출 → 누락분만 추가
└── 미존재 → 전체 신규 생성

Step 6: 테스트 코드 생성
├── 문서 참조: .claude.md → SKILL.md → templates/ → constraints/ → examples/
├── 타입별 매트릭스 적용 (B3): 파라미터/반환 타입별 테스트 패턴
├── 계층별 생성 알고리즘 적용 (templates/ 내 알고리즘)
└── 테스트 파일 생성

Step 7: 검증 및 자동 수정 (B6 의사결정 트리)
├── 컴파일: ./gradlew compileTestJava
│   ├── 성공 → 다음
│   └── 실패 → C1~C4 자동 수정 → 재시도 (최대 3회)
├── 테스트: ./gradlew test --tests "{패키지}.{ClassName}Test"
│   ├── 성공 → 다음
│   └── 실패 → T1~T4 자동 수정 → 재시도 (최대 3회)
└── 커버리지: ./gradlew jacocoTestReport jacocoTestCoverageVerification
    ├── 합격 (라인 80%+, 분기 70%+) → 완료 보고
    └── 미달 → 미커버 영역 추가 테스트 생성

Step 8: 완료 보고
└── "기존 N개 + 신규 M개 = 총 K개, 커버리지 라인 N% / 분기 N%"
```

## 5. 자율 실행 알고리즘 상세 명세

### 5.1. 소스 파일 탐색 알고리즘 (B1)

```
입력: {ClassName} (예: UserService)
출력: 소스 파일 절대 경로

알고리즘:
1. result = Glob("src/main/java/**/{ClassName}.java")
2. if len(result) == 0:
     result = Glob("src/main/**/{ClassName}.java")
3. if len(result) >= 2:
     base_package = read(".claude.md").base_package  # 예: com.nhcard.al.tt
     result = filter(result, contains(base_package_path))
4. if len(result) >= 2:
     present_list_to_user(result)
     return user_selection
5. if len(result) == 0:
     report("파일을 찾을 수 없습니다")
     STOP
6. return result[0]
```

### 5.2. 테스트 수 산출 알고리즘 (B2)

```
입력: 소스 분석 결과
출력: 레벨별 테스트 수

알고리즘:
  total = max(10, public_methods × 3)
  level1 = public_methods           # 각 메서드 최소 1개
  level3 = throw_statements         # 각 throw 문 1개
  level4 = max(2, domain_checks)    # 최소 2개
  level2 = total - level1 - level3 - level4

  if level2 < 0:
    total = level1 + level3 + level4  # 총 수 상향 조정
    level2 = 0
```

### 5.3. 파라미터 타입별 Edge Case 매트릭스 (B3)

| 파라미터 타입 | 테스트 값 | 예상 동작 | 테스트 수 |
|---|---|---|---|
| `String` | `null`, `""` | 예외/기본값 | 2 |
| `Long` | `null`, `0L`, `-1L` | 예외 | 2~3 |
| `Integer` | `null`, `0`, `-1` | 예외 | 2~3 |
| Object (DTO) | `null`, 필수필드 누락 | 예외 | 2 |
| `List<T>` | `null`, `emptyList()` | 예외/빈 처리 | 2 |
| `LocalDate` | `null` | 예외 | 1 |
| `LocalDateTime` | `null` | 예외 | 1 |
| `boolean`/`Boolean` | `true`, `false` | 분기 실행 | 2 |
| `enum` | 각 값 | 값별 분기 | enum 수 |

### 5.4. 반환 타입별 어설션 패턴 (B4)

| 반환타입 | Happy Case | Edge Case |
|---|---|---|
| Object (DTO) | `isNotNull()` + 핵심 필드별 `isEqualTo()` | 파라미터 null → 예외 |
| `List<T>` | `hasSize(N)` + 첫 요소 검증 | `isEmpty()` |
| `void` | `doesNotThrowAnyException()` + `verify()` | 파라미터 null → 예외 |
| `int` | `isEqualTo(1)` | `isZero()` |
| `boolean` | `isTrue()` | `isFalse()` |
| `Optional<T>` | `isPresent()` + `get()` 필드 검증 | `isEmpty()` |

### 5.5. 기존 테스트 자율 처리 (B5)

```
입력: {ClassName}
출력: 추가 모드 또는 신규 모드

알고리즘:
1. existing = Glob("src/test/java/**/{ClassName}Test.java")
2. if existing:
     a. read(existing) → extract @Test methods
     b. compare with source public methods + throw statements
     c. identify uncovered methods/branches
     d. generate ONLY missing tests (preserve existing)
     e. report: "기존 N개 + 신규 M개 = 총 K개"
3. else:
     generate all tests (full creation)
```

### 5.6. 에러 자동 수정 의사결정 트리 (B6)

**컴파일 오류**:

| 코드 | 오류 패턴 | 자동 수정 |
|---|---|---|
| C1 | `cannot find symbol` | import 추가 / 클래스명 오타 수정 |
| C2 | `incompatible types` | Mock `thenReturn()` 반환값 타입 수정 |
| C3 | `package does not exist` | 소스에서 정확한 import 경로 확인 후 수정 |
| C4 | `method does not override` | 메서드 시그니처 소스와 재대조 |
| - | 그 외 | `verification/compile-check.md` 참조 |

**테스트 실패**:

| 코드 | 실패 유형 | 자동 수정 |
|---|---|---|
| T1 | `NullPointerException` | `when()` 설정 누락 → Mock 반환값 추가 |
| T2 | `AssertionError` | 소스 코드 재확인 → 기대값 수정 |
| T3 | `UnnecessaryStubbingException` | 불필요 `when()` 제거 |
| T4 | `InvalidUseOfMatchersException` | Matcher 혼용 수정 (전부 Matcher 또는 전부 리터럴) |
| - | 그 외 | `verification/test-execution.md` 참조 |

**재시도 정책**:
- 각 오류 유형별 최대 3회 자동 수정 시도
- 3회 실패 시 소스 코드 재분석 → 테스트 재생성
- 그래도 실패 시 사용자에게 오류 보고

## 7. CI/CD 통합

### 7.1. GitLab CI 파이프라인

```yaml
# .gitlab-ci.yml (예시)
stages:
  - build
  - test
  - coverage

build:
  stage: build
  script:
    - ./gradlew compileJava compileTestJava

test:
  stage: test
  script:
    - ./gradlew test
  artifacts:
    reports:
      junit: build/test-results/test/*.xml

coverage:
  stage: coverage
  script:
    - ./gradlew test jacocoTestReport jacocoTestCoverageVerification
  artifacts:
    paths:
      - build/reports/jacoco/
  coverage: '/Total.*?([0-9]{1,3})%/'
```
