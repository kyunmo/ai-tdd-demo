# TDD 도입 가이드 설계서

> 작성일: 2026-03-17
> 상태: 확정

---

## 목적

사내 TDD 문화 도입을 위한 실무 가이드 2종 작성.
신입사원 온보딩 수준으로, "왜 도입하는지 → 무엇을 해결하는지 → 어떻게 하는지"를 포함.

## 전제 조건

- 폐쇄망 환경 (농협 내부 설치 AI 모델)
- Claude Code 설치 완료 가정 (설치 가이드는 별도 부서 지원)
- 터미널 명령어 + 출력 예시까지 상세하게 (복붙 가능)

## 문서 구조: 방안 1 — 단계별 여정 (독립 완결형)

각 가이드가 Why → What → How → Next 동일 흐름을 따르되, 상황에 맞게 내용 분기.
두 문서 모두 독립적으로 완결되는 구조.

## 가이드 1: 레거시 프로젝트 TDD 가이드

**파일**: `docs/guides/legacy-tdd-guide.md`
**예시 프로젝트**: ai-tdd-demo (UserService, NoticeService 등)

### 목차

```
1. Why — 왜 테스트 코드가 필요한가
   - 레거시 프로젝트의 현실 (변경 공포, 사이드이펙트, 수동 검증 비용)
   - 테스트 자동화가 해결하는 것
   - AI 에이전트 활용 시 생산성 효과

2. What — 무엇을 할 것인가
   - Phase 1: 기존 소스에 테스트코드 생성 (Test-After)
   - Phase 2: 신규 기능을 TDD로 개발 (Test-First)
   - 최종 목표: 테스트 안전망 구축 → TDD 문화 전환

3. How — 환경 세팅
   - 3.1. 프로젝트에 에이전트 파일 배치 (.claude/agents/)
   - 3.2. 에이전트 역할 및 동작 방식
        - test-generator: 소스 분석 → 4-Level 테스트 자동 생성
        - tdd-review: 생성된 테스트 품질 검증 (100점 채점)
        - 에이전트 간 협업 흐름 (생성 → 리뷰 → 수정)
   - 3.3. 스킬 문서 복사 (docs/ai-tdd-skills/)
        - 스킬 문서 체계 설명 (templates, constraints, verification 등)
        - 에이전트가 스킬 문서를 참조하는 구조
   - 3.4. .claude.md 프로젝트 설정 파일 작성
   - 3.5. 빌드 설정 (JaCoCo, PIT 플러그인)
   - 3.6. 에이전트 동작 확인

4. How — Phase 1: 기존 소스 테스트 생성
   - 4.1. 대상 클래스 선정 (우선순위 기준)
   - 4.2. 단일 클래스 테스트 생성 실습 (UserService 예시)
        - 에이전트 호출 → 생성 → 컴파일 → 실행 → 커버리지
        - 터미널 명령어 + 출력 예시 전체 포함
   - 4.3. 배치 생성 (패키지 단위)
   - 4.4. 리뷰 에이전트로 품질 확인
   - 4.5. 커버리지 목표 달성 확인

5. How — Phase 2: 신규 기능 TDD 개발
   - 5.1. Red-Green-Refactor 사이클 설명
   - 5.2. 시나리오: 기존 프로젝트에 신규 기능을 TDD로 추가
        - Red: 테스트 먼저 작성 (에이전트 활용)
        - Green: 최소 구현
        - Refactor: 리팩토링 + 테스트 재실행
   - 5.3. 전체 흐름 터미널 예시

6. 트러블슈팅 & FAQ

7. Next — TDD 문화 확산
   - 팀 도입 단계별 로드맵
   - 코드 리뷰에 테스트 포함 기준
```

## 가이드 2: 신규 프로젝트 TDD 가이드

**파일**: `docs/guides/new-project-tdd-guide.md`
**예시 시나리오**: 게시판(Board) CRUD를 TDD로 개발

### 목차

```
1. Why — 왜 처음부터 TDD인가
   - "나중에 테스트 작성"이 실패하는 이유
   - TDD가 설계를 개선하는 원리
   - AI 에이전트 + TDD = 생산성과 품질 동시 확보

2. What — 무엇을 할 것인가
   - Red-Green-Refactor 사이클 상세 설명
   - AI 에이전트가 각 단계에서 하는 역할
   - 최종 목표: TDD가 기본인 개발 문화

3. How — 환경 세팅
   - 3.1. 신규 프로젝트 생성 (Spring Initializr 등)
   - 3.2. 에이전트/스킬 문서 배치 (레거시 가이드 3장 참조 + 차이점만)
   - 3.3. 빌드 설정

4. How — 게시판 CRUD TDD 실습
   - 4.1. 요구사항 정의 (게시글 등록/조회/수정/삭제)
   - 4.2. Service 계층 TDD
        - Red: test-generator로 테스트 먼저 생성
        - 컴파일 실패 확인 (터미널 출력)
        - Green: 최소 구현 코드 작성
        - 테스트 통과 확인 (터미널 출력)
        - Refactor: 개선 + 테스트 재실행
   - 4.3. Controller 계층 TDD (같은 사이클)
   - 4.4. Mapper 계층 TDD
   - 4.5. 전체 커버리지 확인 + 리뷰 에이전트

5. How — 실전 팁
   - 어떤 테스트를 먼저 작성할 것인가 (우선순위)
   - AI 에이전트 프롬프트 잘 쓰는 법
   - 테스트가 설계를 이끄는 사례

6. 트러블슈팅 & FAQ

7. Next — TDD 문화 확산
   - 팀 컨벤션 수립
   - CI/CD 파이프라인에 테스트 게이트 추가
```

## 공통 원칙

- **신입사원 온보딩 수준**: 사전 지식 최소 가정
- **Why → What → How → Next**: 동기 부여 → 이해 → 실습 → 확산
- **터미널 예시**: 모든 명령어 + 출력 결과 포함 (복붙 가능)
- **폐쇄망 전제**: Claude Code + 내부 AI 모델 사용
